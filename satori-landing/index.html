Agents Still Build Context the Hard Way

The problem is not access to search. It is how agents assemble context: grep chains, nearby matches, broad file dumps, and guesses about what matters. Real codebases need a better path from intent to implementation. And when an agent makes a decision, you still need to see the evidence it used.

It follows grep chains

Agents search one file, then another, trying to reconstruct the system from fragments instead of navigating a structured map.

It edits the obvious file

Nearby text is not always the implementation that owns the behavior. The obvious match can be a wrapper, test, or stale clue.

It misses related code

A small change can affect callers or callees. Satori surfaces that nearby context when graph support is available.

It trusts stale context

Real repos move while agents work. Satori checks freshness and warns when context needs repair before the agent relies on it.

One Flow: From Intent to Evidence

Satori turns codebase investigation into a fixed path agents can follow: ask by intent, land on symbols, open exact evidence, check related context, then propose changes with less guesswork.

01 Search by intent

Ask what behavior or ownership area matters. Satori retrieves relevant implementation areas from plain English, not just filenames.

02 Navigate by symbols

Move from search results to file outlines, functions, classes, and implementation spans that match how code is structured.

03 Open exact evidence

Read the function, class, or line range that matters instead of flooding the agent with broad file context.

04 Check related context

Inspect callers and callees when supported, with fallbacks instead of guessed graph context.

Satori does not edit your source code. It gives agents better evidence before edits happen in your editor or agent host.

Core Capabilities

Satori is not another semantic search box. It gives coding agents structured repo context they can act on: intent retrieval, symbols, exact reads, related paths, freshness state, and recovery steps.

Plain-English retrieval

Find behavior from intent, not only filenames or exact tokens.

Symbol-aware navigation

Move from search hits to functions, classes, outlines, and implementation spans.

Exact reads

Open exact symbols or line ranges instead of feeding the agent a noisy file dump.

Related code paths

Check nearby relationships when graph support is available for the language and symbol.

Freshness warnings

Detect stale context before an agent relies on old repo state.

Recovery guidance

Return clear next steps when an index is stale, partial, or not ready.

Open Source and Available Today

Satori is usable today: public source, published packages, supported client setup, and a small MCP surface agents can actually follow.

MIT open source

Source is public and inspectable for developers evaluating the tool.

Published packages

The project ships as @zokizuan/satori-core, @zokizuan/satori-mcp, and @zokizuan/satori-cli.

Supported clients

The CLI installer supports Codex, Claude, and OpenCode workflows.

Vector storage

Supports Milvus-compatible vector storage today; embedded local vector storage is planned.

Embedding providers

Supports OpenAI, VoyageAI, Gemini, and Ollama embeddings through runtime configuration.

Six MCP tools

Small by design: index status, search, outlines, reads, graph context, and index management.

Toward Easier Local and Team Workflows

Satori is self-hosted today, with Ollama available for local embeddings and Milvus-compatible vector storage required for indexing. Next work focuses on making setup lighter: embedded vector storage such as Zvec, stronger retrieval quality, broader language coverage, and shared repo context for teams.
